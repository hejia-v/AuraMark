# AuraMark 产品设计文档

版本：v1.1  
日期：2026-03-10

## 1. 产品定义

AuraMark 是一个面向专业写作与技术文档编辑的 **全 WPF 原生 Markdown 编辑器**。它不是“WPF 壳 + WebView2 富文本前端”的组合，而是一个拥有自身编辑器内核的桌面应用。

产品关键目标：

- 原生 WPF 渲染与交互
- Markdown 语义优先
- 数据层与 UI 完全分离
- 所有编辑操作 Action 化
- 可做纯数据层测试
- 支持增量解析、局部布局、局部刷新
- 保持当前产品的专业视觉风格

---

## 2. 背景与问题

当前工程本质上是：

- WPF 桌面壳
- WebView2 承载 Web 富文本编辑器
- 另有源码编辑区域
- 工程中使用了 `AvalonEdit`、`Markdig`、`WebView2`

这种做法的问题：

1. 调试麻烦，维护两套渲染系统与事件系统。
2. WebView2 和 WPF 在焦点、层级、消息桥接、状态同步上天然复杂。
3. 很多行为只能通过集成测试验证，数据层不够干净。
4. 当产品要进化为“可控编辑器内核”时，Web 技术并不是主要价值来源。

---

## 3. 愿景

AuraMark 不是普通富文本编辑器，而是：

> 一个拥有可控文本快照、Markdown 语法树、动作驱动编辑层、可扩展布局系统和原生 WPF Surface 的编辑器内核。

它应当同时具备：

- 桌面工具的稳定性
- IDE 式的结构控制能力
- 优秀的 Markdown 视觉表达能力
- 长期可演进性

---

## 4. 目标用户

主要用户：

1. 程序员、技术写作者
2. 需要大量代码块与结构化内容的作者
3. 长文写作者与知识工作者
4. 重视布局与编辑器手感的专业用户

---

## 5. 价值主张

### 5.1 对用户

- 更稳定的桌面编辑体验
- 更一致的光标、选区和命中测试行为
- 更强的代码块展示能力
- 更安静、耐看的视觉风格
- 更接近专业编辑器而不是网页嵌入器

### 5.2 对产品与开发

- 可维护性更高
- 测试层次更清晰
- 核心逻辑不依赖 UI 控件
- 更适合演进出复杂块级能力与增量更新

---

## 6. 总体产品判断

### 6.1 是否可完全放弃 Web 路线

结论：**可以，而且值得。**

### 6.2 WPF 能否复刻现有视觉

结论：**能。** 当前界面的大多数效果可在 WPF 中高保真复现，包括：

- 圆角卡片
- 半透明背景
- 阴影
- 渐变
- 状态胶囊
- 淡入淡出与位移过渡
- Metadata 卡片
- 标题、段落、引用、列表、表格、行内 code
- 图片圆角阴影
- 代码块卡片、toolbar、gutter

### 6.3 真正的难点

真正的难点不是样式，而是：

- 中文/日文输入法
- 光标与选区
- 折行后的 caret 行为
- 跨 block 选区
- 局部布局与命中测试
- Markdown 增量解析

## 7. 视觉与交互结论

### 7.1 当前视觉方向判断

当前截图已经处于“专业且好看”的区间：

- 三栏布局
- 中间纸面式正文舞台
- 冷静蓝灰配色
- 大量留白
- 代码块卡片化
- 右侧 TOC 清晰

### 7.2 建议保留的视觉元素

1. 左侧文件树 / 中间正文 / 右侧 TOC
2. 冷灰蓝调主题
3. 中间文档纸面感
4. metadata 卡片
5. 代码块卡片化
6. 顶部文档标题 + dirty 状态点

### 7.3 建议优化的地方

1. 中央正文区再白一点，更“纸面化”
2. TOC 层级呼吸感更强
3. 顶部工具条主次更清晰
4. 左树与正文之间层次更柔和

---

## 8. 产品原则

1. **语义优先**：编辑器首先理解 Markdown 结构。
2. **数据优先**：复杂性先在数据层解决。
3. **Action 驱动**：所有操作都可表达为 action。
4. **局部更新**：只刷新必要的区域。
5. **UI 只做显示与输入采集**。
6. **先正确，再精细优化**。

## 10. 交互设计概要

### 输入
所有输入统一映射为 `ReplaceSelectionWithTextAction`。

### 删除
- Backspace：删选区或删前一个字符
- Delete：删选区或删后一个字符

### 回车
第一版只插入换行。后续可扩展：

- 列表续写 / 退出
- 引用续写 / 退出
- 标题断为普通段落
- code fence 内普通换行

### 移动
- 左右：基本移动
- 行首/行尾：基于文本行
- 上下：基于 layout + hit test + PreferredX

## 11. 分阶段交付

### 阶段 1：数据层
- 文本快照
- parser
- reducer
- undo/redo
- 单元测试

### 阶段 2：只读布局与渲染
- paragraph / heading / code fence
- 只读 WPF 渲染

### 阶段 3：交互闭环
- 点击
- 选区
- 输入
- 删除
- 回车
- undo/redo

### 阶段 4：手感增强
- 智能 Markdown 行为
- 更精确 hit test
- 更稳 IME
- 更好上下移动

---

## 12. 验收标准（第一版）

必须成立：

1. 不依赖 WebView2
2. 能显示 markdown 正文
3. 可原生输入、删除、回车、撤销重做
4. 代码块视觉美观
5. 选区与 caret 可用
6. 大部分逻辑可通过纯数据测试验证

可接受的第一版妥协：

1. 命中测试先用近似字符宽度
2. undo/redo 先全量 dirty
3. inline parser 先整块重算
4. code fence 先不做复杂块内增强

---

## 13. 主要风险

高风险项：

- IME
- 精确 hit test
- 跨 block 选区
- 折行与 caret
- grapheme cluster 级删除
- 复杂 Markdown 增量解析

低风险项：

- 视觉还原
- 三栏布局
- metadata 卡片
- 代码块卡片
- heading / paragraph 渲染

---

## 14. 产品结论

1. 全 WPF 路线：**可行**
2. 视觉高保真复刻：**可行**
3. `SourceText` / `TextSpan` 风格文本内核：**非常合理**
4. Action 驱动、数据/UI 分离、局部更新：**非常适合编辑器**
5. 先纯数据测试再 GUI 测试：**是正确顺序**

> 最难的不是画界面，而是拥有编辑器内核。  
> 但 AuraMark 这个方向值得这么做。
